El problema

¿Por qué existe Korva?

😤 Sin Korva

  • 🔴 El agente IA empieza desde cero en cada sesión — no sabe nada del proyecto
  • 🔴 Importa infraestructura desde el dominio sin saberlo — viola la arquitectura hexagonal
  • 🔴 El mismo bug se resuelve múltiples veces por distintos developers
  • 🔴 El AI codifica directamente sin planificar — no hay especificación
  • 🔴 Secretos hardcodeados pasan el code review sin detección
  • 🔴 El conocimiento del equipo vive solo en Slack/Notion — el AI jamás lo ve

✅ Con Korva

  • 🟢 El AI carga el contexto automáticamente al inicio de cada sesión
  • 🟢 El pre-commit bloquea violaciones de arquitectura antes del commit
  • 🟢 Cada bug resuelto queda guardado — no se repite
  • 🟢 SDD obliga a explorar → especificar → diseñar antes de codificar
  • 🟢 SEC-001 detecta secretos hardcodeados en tiempo real
  • 🟢 Los Scrolls inyectan conocimiento de arquitectura al AI automáticamente
Arquitectura

6 componentes, 1 ecosistema

🧠

Vault

Motor de memoria persistente. SQLite + FTS5. 18 herramientas MCP. Expone REST API en :7437

🛡️

Sentinel

Validador de arquitectura. Hook pre-commit. Reglas HEX, NAM, SEC, TEST. Perfiles minimal / standard / strict

📚

Lore

Inyección de conocimiento. Los "Scrolls" se auto-cargan al abrir archivos. 18+ guías curadas + privadas por equipo

🔨

Forge / SDD

Workflow Spec-Driven. 9 fases: explore → propose → spec → design → tasks → apply → verify → archive → onboard

⌨️

CLI

Comando korva. 18 subcomandos. Setup automático de VS Code, Claude Code, Cursor con korva setup --all

🌐

Beacon

Dashboard web React 19. Explorer de observaciones, sesiones, Lore Manager, panel admin. Mismo servidor :7437

Integración VS Code

Todo conectado en un solo editor

⚡
Un solo comando instala todo:
korva setup --all
Auto-configura VS Code, Claude Code y Cursor.

Lo que ocurre en tu sesión VS Code

1 Abres un archivo → Lore detecta el tipo y auto-carga el Scroll de arquitectura correspondiente
2 El AI recibe contexto → Vault inyecta decisiones previas, bugs resueltos y patrones del proyecto
3 Haces un commit → Sentinel valida arquitectura y bloquea si hay violaciones
4 El AI aprende → vault_save guarda la decisión para la próxima sesión

Componente 1 — Vault

🧠 La memoria persistente del equipo

¿Qué guarda?

decision pattern bugfix learning antipattern feature refactor discovery task
🔐
Privacy Filter automático: antes de guardar cualquier observación, el filtro elimina tokens, claves API, rutas de archivo, PII y datos sensibles.

📁 Los datos viven en ~/.korva/vault.db
🌐 REST API en localhost:7437
🔌 18 herramientas MCP via protocolo stdio

Ejemplo real: guardar una decisión

// El AI hace esto automáticamente después de resolver un bug
vault_save({
  type:    "bugfix",
  title:   "Race condition en procesador de pagos",
  content: "Dos requests concurrentes pueden
           cobrar doble. Fix: Redis distributed
           lock en payment_id con TTL 30s.",
  project: "payments",
  tags:    ["concurrency", "redis"]
})
// → { "id": "01HX9Z...", "status": "saved" }
🔍
¿Por qué sirve? La próxima vez que alguien trabaje en payments, el AI cargará este bugfix automáticamente y no repetirá el mismo error.

Vault — Dry Run

Previsualizar antes de guardar

🔎
Con dry_run: true puedes ver exactamente qué guardará el Vault después de aplicar el filtro de privacidad, sin escribir nada en la base de datos. 
// Previsualizar qué pasaría con un vault_save
vault_save({
  type:    "decision",
  title:   "API Key de Stripe: sk_live_4xABC...",
  content: "Usamos Bearer token ABC123 para auth",
  project: "payments",
  dry_run: true
})
// Respuesta — el filtro actuó:
{
  "dry_run":          true,
  "would_save":       true,
  "filtered_title":  "API Key de Stripe: [REDACTED]",
  "filtered_content": "Usamos Bearer [REDACTED] para auth",
  "title_changed":   true,
  "content_changed":  true
}
✅
El filtro elimina automáticamente tokens, API keys, JWTs, rutas absolutas y PII antes de persistir cualquier dato. Nunca se almacenan secretos.
Componente 2 — Sentinel

🛡️ Guardián de arquitectura en pre-commit

Reglas incluidas

HEX-001Domain no puede importar infraestructura
HEX-002Application no puede importar infraestructura
HEX-003Sin console.log en archivos de producción
SEC-001Sin secretos hardcodeados
NAM-001DTOs deben terminar en DTO (no Dto)
NAM-002Port tokens en SCREAMING_SNAKE_CASE

Perfiles de reglas

minimal Solo SEC-001 — para equipos nuevos
standard HEX-001/002/003 + SEC-001 — por defecto
strict Todas las reglas — compliance total
pre-commit hook
$ git commit -m "feat: add payment service" korva-sentinel analyzing 3 files... ERROR src/domain/insurance.entity.ts:14 [HEX-001] Domain imports from infrastructure. Found: import from '../infrastructure/db/...' Domain must only depend on its own types. ERROR src/application/services/payment.ts:3 [SEC-001] Potential hardcoded secret. Found: apiKey = 'sk_live_4xABC123...' Use process.env.* or configService.get() Files: 3 Errors: 2 Warnings: 0 ✗ Commit blocked — fix errors before committing
⚙️
JSON para CI/CD:
korva-sentinel --format json --profile strict src/...
Integra con GitHub Actions, GitLab CI, Jenkins.
Sentinel — Código en acción

Lo que Sentinel detecta y cómo corregirlo

❌ ANTES — commit bloqueado 
src/domain/entities/insurance.entity.ts// HEX-001: El dominio NO debe importar infra
import { InjectRepository } from
  '@nestjs/typeorm';            // 💥 violación

export class InsuranceEntity {
  id: string;
  premium: number;
}

// SEC-001: Secreto hardcodeado
const apiKey = 'sk_live_4xABC123XYZ';   // 💥 violación
✅ DESPUÉS — commit permitido 
src/domain/entities/insurance.entity.ts// Dominio limpio — solo tipos propios
export class InsuranceEntity {
  id: string;
  premium: number;
}

// Secreto en variable de entorno
const apiKey =
  configService.get('STRIPE_API_KEY'); // ✅
🎯
Sentinel garantiza que la arquitectura hexagonal se mantenga intacta a medida que el equipo escala. El AI que genera código también pasa por el mismo gate.
Componente 3 — Lore

📚 Conocimiento de arquitectura inyectado automáticamente

📖
Los Scrolls son guías de arquitectura que el AI recibe automáticamente al abrir archivos que coinciden con los patrones configurados. Sin Scrolls, el AI propone código genérico. Con Scrolls, sigue tus convenciones exactas.

Scrolls curados disponibles

nestjs-hexagonal typescript-patterns stripe-payments docker-kubernetes react-components testing-standards claude-api mcp-builder go-patterns + 60 más

Scrolls privados del equipo

Tu equipo puede crear Scrolls privados con convenciones internas (cómo estructurar módulos, reglas de naming propias, etc.) que viven en un repo privado — nunca en el repo público.

Estructura de un Scroll

lore/curated/nestjs-hexagonal.md---
name: nestjs-hexagonal
triggers:
  - "src/**/*.entity.ts"
  - "src/**/domain/**"
---

## Regla 1: El dominio es la fuente de verdad

### Correcto ✅
export class InsurancePolicy {
  constructor(private readonly premium: number) {}
  validate(): boolean { return this.premium > 0; }
}

### Incorrecto ❌ — no importar TypeORM en domain
import { Column } from 'typeorm';
Componente 4 — Forge / SDD

🔨 Workflow estructurado: del problema al código

Spec-Driven Development (SDD) es un workflow de 9 fases que obliga al AI a explorar y especificar antes de escribir una sola línea de código.

explore
→
propose
→
spec
→
design
→
tasks
→
apply 🔒
→
verify 🔒
→
archive
→
onboard

Ejemplo: avanzar a fase apply

// 1. Pedir el checklist de calidad
vault_qa_checklist({
  phase: "apply",  language: "typescript"
})
// → 12 criterios: tests, patterns, e2e, style, security...

// 2. El AI evalúa y guarda el checkpoint
vault_qa_checkpoint({
  project: "payments", phase: "apply",
  status:  "pass",     score:  87,
  gate_passed: true
})
// → { gate_unlocked: "apply → verify desbloqueado" }
🔒
Las fases apply y verify están gateadas: el AI no puede avanzar sin un checkpoint con gate_passed: true y score ≥ 70. Esto garantiza calidad antes de mergear.
📊
vault_sdd_phase({ project: "payments" })
Muestra en qué fase está el proyecto actualmente. El AI siempre sabe dónde está parado.

Privacidad y seguridad

3 reinos de privacidad

🌍

Reino 1 — Público

Korva open source en GitHub (MIT). El código base del proyecto. Nunca contiene datos de equipos ni observaciones.

github.com/alcandev/korva
🏢

Reino 2 — Privado del equipo

Repo privado de la organización con Scrolls internos, reglas de Sentinel personalizadas e instrucciones AI del equipo.

github.com/YOUR-ORG/team-profile
💻

Reino 3 — Tu máquina

El Vault local. Todas las observaciones del proyecto viven en ~/.korva/vault.db. Solo se sincronizan si tú decides activar Hive.

~/.korva/ (100% local)
🔐
admin.key con permisos 0600. Nunca se serializa ni loguea. Solo se lee de ~/.korva/admin.key. RS256 JWS para autenticación de admin.
🛑
Kill-switch de Hive: KORVA_HIVE_DISABLE=1 deshabilita toda la pipeline de sincronización saliente. La inaccesibilidad de Hive nunca bloquea el flujo local.
